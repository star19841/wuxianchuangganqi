# Soul - 项目灵魂

> 本文件由 AI 维护。
> 用途：定义本项目在长期商业化开发中的工程气质、价值排序与反模式边界。
> 每次开始编码前阅读本文件，确保实现不仅“能运行”，更符合项目长期演进方向。

---

## 一、这不是一个练手仓库，而是一个商业产品骨架

这个项目存在的意义，不是为了快速拼出几个页面，也不是为了展示某种技术姿态。
它的意义在于：以足够清晰、足够克制、足够稳定的方式，构建一个能长期承载业务迭代的 Web 产品基础工程。

这里的“商业”不等于复杂术语，不等于层层包装，也不等于一开始就堆满中间件和基础设施。
这里的“商业”意味着：

- 代码最终要服务业务目标
- 功能最终要承受持续修改
- 问题出现时必须能定位
- 人员切换时必须能接手
- 规模增长时不能轻易失控

---

## 二、分层不是形式，而是控制复杂度的手段

MVC 在这个项目里不是目录摆设，而是一种复杂度控制方法。

如果边界清楚：

- Controller 负责请求与响应边界
- Model 负责数据与业务规则
- View 负责呈现与交互承载

那么项目在扩展时，复杂度会被限制在各自层内。

如果边界模糊：

- Controller 会膨胀成业务黑洞
- Model 会退化成散乱 SQL 容器
- View 会偷偷承接本不属于它的决策逻辑

商业项目最怕的不是某一次功能写慢，而是长期迭代后每次改动都牵一发动全身。
所以，守住边界，本质上是在守住未来的开发速度。

---

## 三、真实性比“像大厂项目”更重要

这个项目拒绝伪成熟。

如果当前只有 SQLite，就如实写 SQLite。
如果当前只有基础认证流，就如实承认它还是基础认证流。
如果某些能力只是未来规划，就明确写成演进方向，而不是包装成当前能力。

文档、代码、目录、描述必须对齐。
一个真实但朴素的项目，远比一个靠措辞营造成熟幻觉的项目更可靠。

商业项目的可信度，来自一致性，不来自话术。

---

## 四、可维护性高于短期炫技

本项目不追求“写法看起来很厉害”，而追求“团队长期接得住”。

这意味着：

- 命名要明确
- 控制流要直白
- 结构要稳定
- 抽象要克制
- 代码要让后来的开发者可以在有限时间内建立理解

本项目拒绝以下倾向：

- 把简单问题包成复杂框架
- 为了显示架构感过早引入大量抽象
- 用技巧压过可读性
- 用想象中的未来场景主导今天的工程决策

真正成熟的代码，往往不是最花哨的，而是最容易持续维护的。

---

## 五、稳定性、安全性、可排障性都是产品能力

商业 Web 项目交付的从来不只是页面或接口。
它交付的是“用户可用、团队可控、问题可查”的整体能力。

因此，这个项目天然要求：

- 输入校验不是装饰，而是入口防线
- 密码安全不是附加项，而是底线
- 错误处理不是 try 一下就结束，而是要兼顾用户体验与排查能力
- 日志不是可有可无，而是未来运营、维护、追踪问题的重要依据

如果一个功能在 happy path 下能跑，但失败时无从定位、出现异常时无人能解释、配置变化时轻易崩掉，那么它就不算完成。

---

## 六、小步演进，是商业项目里最可靠的速度

这个项目不鼓励以“重写一遍会更优雅”为理由频繁推倒重来。
真正高质量的商业工程，通常来自连续的小步改良，而不是几次情绪化的大重构。

应当优先采用的方式是：

- 沿现有结构扩展
- 在真实痛点出现时再引入新层次
- 每次改动都尽量保持落点清晰
- 在保证当前行为稳定的前提下逐步提升结构质量

小步演进并不保守。
它是在确保业务持续交付的同时，让架构真正变好。

---

## 七、AI 在这个项目里应具备的气质

AI 不应像一个只会套模板的代码生成器。
AI 应该像一个懂商业项目节奏的工程协作者。

应有的气质：

- 先观察，再修改
- 先对齐现状，再提出演进
- 先解决真实问题，再考虑抽象升级
- 先保证结构清晰，再追求形式上的“完整”

不应出现的行为：

- 幻觉式补全不存在的基础设施
- 忽视现有项目结构，强行套入通用脚手架模式
- 为了显得专业而引入并不必要的复杂度
- 用含糊的文档和术语掩盖当前状态其实还没到那一步

---

## 八、本项目欢迎什么样的代码

本项目欢迎这样的代码：

- 简洁但不草率
- 明确但不僵硬
- 稳定但不保守
- 便于阅读，也便于继续扩展
- 在前端界面实现上尊重既定技术体系，而不是随意更换组件生态

看到一段好的代码，团队应当产生这样的感受：

- 我知道这段代码属于哪一层
- 我知道后续改动应该从哪里下手
- 我知道它为什么这样设计
- 我不需要猜测它背后的隐含规则

## 九、本项目拒绝什么

### 9.1 伪企业化

拒绝为了“看起来像商业项目”而制造一堆空壳结构。

例如：

- 没有真实编排需求却先建 `services/`
- 没有实际落地却在文档里默认存在 CI、监控、审计、配置中心
- 用过度宏大的描述掩盖项目仍处于基础骨架阶段的事实

### 9.2 控制器膨胀

拒绝让 Controller 同时承担：

- 参数处理
- 业务判断
- 数据访问
- 页面流程
- 异常兜底的全部复杂度

一旦 Controller 膨胀，后续所有业务都会开始失控。

### 9.3 模板偷走业务逻辑

拒绝把 View 当成逻辑层。

模板负责展示，不负责决定业务规则。
前端交互可以增强体验，但不应成为核心约束的唯一来源。

### 9.4 过早抽象

拒绝为了还没发生的未来提前支付复杂度。

今天只有一个明确场景，就先把这一个场景写清楚、写扎实。
当第二个、第三个真实场景出现时，再决定是否抽象。

### 9.5 文档表演化

拒绝把文档写成“看上去很高级”的装饰材料。

文档必须帮助后来者理解：

- 当前有什么
- 为什么这么做
- 下一步应该怎么接

而不是帮助项目显得比实际更成熟。

### 9.6 前端体系漂移

拒绝在没有明确任务要求的情况下，随意偏离当前前端技术基线。

当前项目前端已明确采用 `layui-v2.13.7`，并已在 `dist/layui-v2.13.7/` 放置本地组件资源。
因此后续界面开发应优先在该体系内完成，而不是临时切换到其他 UI 框架、在线 CDN 方案或自造一套风格体系。

---

## 十、这个项目认为什么是“好”

一个好的变更，通常具有以下特征：

- 它符合当前 MVC 方向
- 它提升了业务承载能力，而不是只增加表面复杂度
- 它不会让结构更浑浊
- 它能被开发者用几句话解释清楚
- 它为后续商业化扩展留出了空间，但不提前透支复杂度

一个好的商业项目，不是“什么都先准备好了”的项目。
而是“每一步都清楚自己为什么这样长出来”的项目。

---

## 十一、最终提醒

这个项目的灵魂可以浓缩成一句话：

**用克制而真实的工程方式，构建一个能持续承载业务增长的 Tornado Web 产品骨架。**

当存在取舍时，优先选择：

- 真实，而不是包装
- 清晰，而不是炫技
- 边界，而不是省事
- 稳定，而不是冲动重写
- 可维护，而不是短期看起来更快
