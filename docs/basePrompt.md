## 一、项目定位

本项目是一个基于 **Python + Tornado** 构建的 **MVC 三层架构 Web 应用骨架**，目标不是完成一次性的功能演示，而是作为一个可持续迭代的 **商业级通用 Web 产品基础工程**。

项目需要具备以下特征：

- 支撑后续持续增加业务模块
- 保持清晰的分层边界与可维护性
- 在小规模起步的同时，为后续商业化扩展预留合理演进空间
- 兼顾页面渲染型 Web 场景与未来 API 化能力
- 以稳定性、安全性、可排障性和一致性为优先目标

本项目不是：

- 一次性脚本集合
- 只追求“能跑”的练手仓库
- 靠堆砌抽象和术语包装出来的伪企业化工程

## 三、商业级项目目标

后续所有实现都应围绕“商业项目基础工程”来推进，而不是围绕“快速拼出页面”来推进。

核心目标包括：

- **可维护**：后续开发者能快速理解代码结构与职责边界
- **可扩展**：可以继续增加业务模块、权限规则、配置能力和接口能力
- **可运营**：关键行为应具备日志、错误信息、排障入口
- **可演进**：允许从当前 SQLite 起步，逐步升级到更完整的商业化部署形态
- **可约束**：AI 与人工协作时有统一的分层、命名、交付标准

## 四、AI 在本项目中的角色

AI 在本项目中的职责，不只是“生成代码”，而是作为一个遵守工程纪律的协作开发者。

AI 必须：

- 优先遵守现有 MVC 结构
- 在真实目录与现有模式基础上扩展，而不是另起一套风格

- 把尚未实现的目录、模块、配置系统、部署体系写成当前事实

## 五、MVC 三层边界

这是本项目最重要的硬约束之一。
任何新增功能、重构或修复，都必须先服从边界，再考虑实现细节。

### 5.1 Controller 层：`app/controllers/`

职责：

- 接收 HTTP 请求
- 解析输入参数
- 做基础输入校验
- 调用 Model 层方法
- 决定渲染页面、返回结果或跳转路径
- 将业务结果转为 HTTP 响应

允许：

- 编写 `get()`、`post()` 等请求处理方法
- 使用 Tornado 提供的参数读取能力
- 进行登录校验、跳转控制、渲染控制
- 组织页面流转或接口输出格式

禁止：

- 直接书写 SQL
- 在 Controller 中沉淀核心业务规则
- 将复杂数据处理长期堆积在 Handler 内
- 让 Controller 承担“业务中枢 + 数据中枢”的双重职责

### 5.2 Model 层：`app/models/`

职责：

- 封装数据访问
- 编写查询与持久化逻辑
- 承担业务规则和数据完整性约束
- 负责密码校验、记录查询、状态判断等领域逻辑

允许：

- 编写 SQL
- 处理数据库连接与持久化逻辑
- 编写 Repository 风格的数据访问方法
- 在不引入 HTTP 细节的前提下实现业务规则

禁止：

- 依赖 Tornado RequestHandler
- 调用 `render()`、`redirect()` 等 HTTP 行为
- 直接依赖模板结构或页面跳转语义

### 5.3 View 层：`app/templates/` + `app/static/`

职责：

- 展示数据
- 承载用户输入界面
- 完成轻量交互和页面结构组织

允许：

- 模板继承
- 简单条件展示
- 页面结构与表单渲染
- 必要的 CSS / JS 页面层交互

禁止：

- 编写复杂业务判断
- 把模板变成业务规则载体
- 在前端层偷偷复制后端核心约束并以其为唯一依据

## 七、商业项目编码规则

### 7.1 一般规则

- 一致性优先于炫技
- 明确性优先于“巧妙”
- 小步演进优先于大幅重写
- 局部可理解优先于过早抽象
- 与现有代码风格对齐优先于套用通用模板

### 7.2 业务扩展规则

- 新业务页面：通常对应新增或扩展 Controller、模板和相关 Model 方法
- 新数据实体：优先在 Model 层新增对应 Repository 或数据访问方法
- 新共享请求逻辑：优先考虑是否适合进入 `BaseHandler`
- 新共享业务编排：当 Model 层开始出现明显跨实体编排复杂度时，再考虑引入 Service 层

### 7.3 前端实现规则

- 后续正式界面开发默认采用 `layui-v2.13.7`
- 优先引用本地 `dist/layui-v2.13.7/` 资源，不默认切换到其他 UI 框架
- 页面结构、表单、弹层、表格、导航等通用后台界面能力，优先基于 Layui 现成模式构建
- 如需补充样式或交互，应在尊重 Layui 体系的前提下做克制增强，而不是推翻重来

### 7.4 页面与 API 并行规则

本项目当前以页面渲染为主，但未来可以演进出接口能力。

规则如下：

- 页面型功能优先保持 Tornado 模板渲染模式
- 当出现明显前后端分离需求时，再引入 API 路由体系
- API 一旦引入，应统一命名与版本前缀，例如 `/api/v1/...`
- 页面响应与 API 响应不要混用风格，避免同一路径承担模糊职责

### 7.5 文档真实性规则

- 当前存在什么，就写什么
- 未来建议什么，就标记为“建议”或“演进方向”
- 不得使用模糊表述伪装项目成熟度
- 文档必须服务于后续开发协作，而不是服务于包装感

### 7.6 依赖控制规则

- 优先使用标准库
- 其次使用 Tornado 现有能力
- 只有在明显降低复杂度、风险或重复劳动时，才引入第三方依赖

## 八、配置、安全、日志与异常基线

### 8.1 配置基线

当前项目中部分配置仍在入口文件中直接定义，这是现状，不是终态。

后续商业化演进应遵守：

- 敏感配置优先走环境变量
- 运行参数应逐步从硬编码迁移为可配置
- 不同环境的差异不应靠散落常量维持

### 8.2 安全基线

AI 必须维护或增强以下安全属性：

- 密码不得明文存储
- SQL 必须保持参数化
- 登录保护页面必须保持认证约束
- XSRF 保护不能被随意移除
- 敏感配置不应继续向生产场景硬编码扩散

当前事实说明：

- `cookie_secret` 当前仍硬编码在 `app.py` 中

这表示“当前状态”，不表示“推荐做法”。

### 8.3 日志与排障基线

商业项目不能只在“运行成功时”可见，失败时也必须可定位。

后续实现应逐步满足：

- 关键路径有可读日志
- 关键异常有明确信号
- 错误不能只靠页面报错字符串临时拼接
- 重要行为未来应可扩展为审计或运营排查入口

### 8.4 异常处理基线

- 不允许把异常处理理解为“捕获后吞掉”
- 对用户可见结果要清晰
- 对开发和运维可排查信息要可追踪
- 异常策略应服务于稳定交付，而不是掩盖问题
